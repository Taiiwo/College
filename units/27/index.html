Web Server Scripting Languages ------------------------------ There are many different languages that can be used for server-side scripting as there are many different web server softwares allow for different languages and interpreters to control output of files. This basically means that any language that can output text can also be used for web server scripting. However, due to the syntax and inbuilt functions of certain languages, there are languages, with the inclusion of languages that were built specifically for server side scripting, there exists a few languages that are commonly used over the rest: - PHP - PHP: Hypertext Pre-processor - ASP.NET - Active Server Pages - Python - Perl
PHP - PHP: Hyper Pre-processor ------------------------------ PHP is an open-source programming language that is written specifically for web servers. In syntax, it is very similar to ASP.NET, but with the added functionality and flexibility of it being compatible with almost all web servers, and the inherent advantages of it being open-source.
ASP.NET - Active Server Pages ----------------------------- ASP.NET was a scripting language created by the Microsoft Corporation. It was originally added to Microsoft's IIS server as an addon, but is now a standard component in the build.
Python ------ Python is an interpreted language that was created as a high level scripting language, not a web server scripting language. Using pre-made webservers written in Python, or using existing web server software such as Apache or nGinX, it is possible for you to use mods to execute the Python scripts.
Perl ------ Using pre-made webservers written in Perl, or using existing web server software such as Apache or nGinX, it is possible for you to use mods to execute the Perl scripts. Perl is an interpreted language that was created as a high level scripting language, not a web server scripting language.
Static vs. Dynamic ------------------------------ ### *Static* Static webpages are any type of webpage that doesn't move. The best way to identify a static webpage is to imagine printing it, and ask yourself if you'd lose any of the page's functionality. Static webpages are good for things like information-only sites. They have low resource usage as they run no code on either the server-side, or the client-side. Static webpages are written entirely in HTML and CSS. ### *Dynamic* Dynamic webpages are webpages that have content that changes dynamically. The information on the webpage can be different on each page load, or even while the page is being displayed. This allows web developers to add content to websites that makes websites more interactive, such as having slide-in navigation bars, or, I don't know, something like a webpage you can navigate entirely with the arrow keys ;)
Client-Side vs. Server-Side --------------------------------------- ### *Server-Side* Server-side scripting allows programs to modify the contents of the HTML that is sent to the client-side. This can be useful as the code is never seen by the client, and therefore they cannot see how the data on the page has been put there. You will see this a lot with companies that don't want you to be able to see their methods. Another advantage of server-side scripting is that secure functions may be run. Things like accessing databases and running high priority software on the server can be done from server-side languages. This is useful when you don't want the client to have access to certain functions for security reasons. ### *Client-Side* Client-side code is code that runs inside the browser. It does not need the page to reload to execute, and therefore it can make the page dynamic and responsive. There have been developed many pre-written libraries for doing all sorts of amazing things. For example, behind this text, you can see the output of Trianglify.js. This dynamically creates large SVG images for use as page backgrounds, and significantly speeds up page loads over using large images as backgrounds.

Security Issues --------------- Using server-side scripting opens some large holes in security, and pushes into some controversial ground in terms of ethics.
Server-Side Security Issues --------------------------- Server-side code has comparatively and infinitely higher level of authority over the server machine when compaired to client-side. For this reason, when we look a the drawbacks of server-side scripting, it hold a considerable threat in respect to the heightened possibility of security vulnerabilities due to improperly sanitised user-input. Server-side code almost always takes in data from the user, and if this data is not handled properly, it may be possible to format the data in a way that is able to take over the server-side interpretor, giving an attacker some access to the server machine.
Issues in Server-Side Ethics ---------------------------- When dealing with server-side scripting, it cannot be ignored that a user's data must be passed through a centralised source before it is used. This lowers the privacy of the users' information, as the data could be logged or sold on. There is the further dispute that the user is unable to see the code that is executed server-side. Unlike client-side code, which can be read and debugged using the developer tools in your browser; server-side code cannot be viewed in any such way, and thus restricts the user's confidence in the webserver.
